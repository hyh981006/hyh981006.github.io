热更新基础知识

1.热更新的概念：

游戏或者软件更新时无需重新下载客户端进行安装，而是在应用程序启动的情况下，在内部进行的资源或者代码更新；

2.热更新的好处：

迅速修复Bug—–避免溪红心下载安装包，游戏内部更新修复Bug；
提升玩家留存率—–避免玩家因为超大的安装包而流失；
迅速换“内核”—–挂羊头卖狗肉；

AB包相关知识

AB包的含义：

特定平台的资产压缩包，有点类似于压缩文件，资产包括：模型，贴图，预制体，音效，材质球等等；

AB包的作用：

（1）相对于Resources下的资源，AB包更好的管理资源
【1】Rescoures在打包时定死，只可读，无法进行修改
【2】存储位置是可以自定义的，压缩方式也同样是自定义的，后期可以动态更新；
（2）减少包体的大小
【1】压缩资源
【2】减少初始包大小

（3）热更新
资源热更新：
脚本热更新：
(1)客户端自带很少的默认资源=>（第一步获取资源服务器地址）=>服务端
(2)资源服务器：资源对比文件，最新的各种的AB包，=>通过资源对比文件，检测哪些需要更新，哪些需要下载，下载AB包=>客户端

生成AB包：

（1）Unity编辑器开发，自定义打包工具；
（2）官方提供好的打包工具，Asset Bundle Browser

（2）整完后会在你的Windows/Asset Bundle Browser里看到相应的信息

（3）然后在Asset Bundle Browser里面点击Bulid【相关参数含义如下】
【1】Bulid Target是指发布的平台，具体可支持如下;

【2】Output Path是输出的路径;
【3】Clear Folders：判断是否在每次发布AB包时清空相应的文件夹数据;
【4】Copy to StreamingAssets是指是否在打包的时候相关信息赋值到该文件夹内,是否拷贝到StreamingAssets
【5】Compression是指压缩方式【默认为LZMA压缩方式，即Standard Compression】
主要的信息：
{1}.No Compression无压缩
{2}.Stand Compression（LZMA）压缩是压缩到最小，但是同样所要解压缩也是最慢的，同样如果用某一样资源，需要解压所有
{3}.Chunk Based Compression(LZ4)相比较第二种，压缩大小相对于较大一些，但是最大的好处是能够实现用什么解压什么;

Exclude Type Infomation 在资源包中不包含资源的类型信息

Force Rebulid 重新打包时，需要重新构建包，与Clear Folders不同，不会删除不再存在的包

Ignore Type Tree Changes 增量构建检查时，忽略类型树的更改

Append Hash 将文件哈希值附加到资源包名上

Strict Mode 严格模式，如果打包时报错了，打包失败无法成功【建议勾选】
索要注意的是：在AssetBundles在打包时是不会打包出去的
